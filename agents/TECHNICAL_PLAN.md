# Technical Plan: The Diaries of James David Smillie — Web Application

## 0. Key Facts from the Codebase

| Fact | Detail |
|---|---|
| Total images | 8,210 across 45 years (1865–1909), all in `data/YYYY/` |
| Transcriptions done | 1,887 files across 13 years (1865–1877), in `transcriptions/YYYY/` |
| Transcription artifact | Every `.md` file opens with a spurious fenced code block containing its own path; must be stripped |
| Frontmatter parser | No PyYAML in venv; frontmatter parsed with regex (structure is simple and consistent) |
| `## Left Page` / `## Right Page` | Present in every transcription file, consistently structured |
| Image size | Average 871 KB, ~2000×1800 px |
| Sequence source | `mets.json` exists for all 45 years; `orderLabel` (1-based integer string) is authoritative sequence |
| Manifest size | 329 KB raw / ~131 KB gzipped (flat id→fileUri dict) |
| Transcription text total | ~2.9 MB for 1,887 files; ~13 MB projected for full corpus |
| Unique diary dates (current) | ~3,499 (1865-01-01 to 1877-12-29) |
| Tooling | Python 3.14, Node 25.6, npm 11.8; no `package.json` or `site/` dir yet |

**Key architecture decisions:**

1. **`site/public/data/`** — Generated data files go here, not `dist/`. Vite's
   `publicDir` copies `site/public/*` to `dist/` during build. This avoids the
   `emptyOutDir` conflict where Vite would delete generated files.

2. **`about.html`** (flat file, not `about/index.html`) — Avoids CloudFront
   routing issues with `/about` vs `/about/`. The URL `/about` or `/about.html`
   both work directly.

3. **Images are never in `dist/` locally.** They are uploaded separately to S3
   at `s3://[bucket]/images/YYYY/STEM.jpg` via `make upload-images`.

## 2. Data Files Produced by `build_site.py`

### `site/public/data/manifest.json`

A flat JSON object mapping image IDs to Smithsonian file stems. This is the
only data needed for navigation — everything else is derivable from the ID or
fetched on demand.

```json
{
  "1865/0001": "AAA-AAA_smilsmil_2303576",
  "1865/0002": "AAA-AAA_smilsmil_2303577",
  "1865/0003": "AAA-AAA_smilsmil_2303578",
  ...
  "1909/0126": "AAA-AAA_smilsmil_2311782"
}
```

**Size:** 329 KB raw / ~131 KB gzipped.

**Design note:** `hasTranscription` is intentionally absent. The viewer fetches
the transcription JSON and handles a 404 gracefully by showing a placeholder.
At full corpus launch all images will have transcriptions — no 404s in
production. During local dev with a partial dataset, the image loads
immediately and the transcription area shows a spinner until the fetch
resolves; a 404 replaces the spinner with a placeholder message.

### `site/public/data/year-index.json`

Maps each year to its starting index in the manifest's insertion order and the
count of images for that year. Used by the scrubber (year label positions) and
the nav indicator ("1865 · 1 of 108").

```json
{
  "1865": { "start": 0,    "count": 108 },
  "1866": { "start": 108,  "count": 181 },
  "1867": { "start": 289,  "count": 190 },
  ...
  "1909": { "start": 8084, "count": 126 }
}
```

**Size:** ~1.5 KB.

### `site/public/data/date-index.json`

Maps ISO date strings to manifest IDs. Pre-computed at build time from
transcription frontmatter. Used by the date picker.

```json
{
  "1865-01-01": "1865/0003",
  "1865-01-02": "1865/0003",
  "1865-01-03": "1865/0004",
  ...
}
```

When a date appears on two images, the lower sequence number wins.
**Size:** ~200 KB at full corpus.

### `site/public/data/transcriptions/YYYY/STEM.json`

One file per transcribed image. Contains the transcription split into left and
right page Markdown, plus display metadata.

```json
{
  "id": "1865/0003",
  "dates": ["1865-01-01", "1865-01-02"],
  "sections": {
    "left":  ["almanac"],
    "right": ["diary"]
  },
  "left":  "### November, 1865 — Moon's Phases\n\nFull Moon 3d. 3h. 7m. M.\n...",
  "right": "### January, Sunday 1, 1865\n\nPainting on little oil sketch..."
}
```

The `left` and `right` fields contain raw Markdown (everything after the
`## Left Page` / `## Right Page` headings, with those headings removed). The
browser renders each with `marked.js` and displays them in a two-column layout
mirroring the physical diary spread.

**Size per file:** 1–5 KB. Fetched on demand during navigation.

## 3. Build Pipeline

```
python build_site.py  →  npm run build  →  npx pagefind  →  (deploy)
```

### Step 1: `python build_site.py`

**Inputs:** `data/*/mets.json`, `transcriptions/*/*.md`

**Outputs:**
- `site/public/data/manifest.json`
- `site/public/data/year-index.json`
- `site/public/data/date-index.json`
- `site/public/data/transcriptions/YYYY/STEM.json` (one per transcribed image)
- `pagefind-source/YYYY/STEM.html` (one per diary page, for search indexing)

**Algorithm:**

1. For each year 1865–1909, load `data/YYYY/mets.json`. Iterate entries in
   `orderLabel` order to build the ordered image list.

2. For each image, attempt to read `transcriptions/YYYY/STEM.md`. If it exists:
   - **Strip artifact:**
     ```python
     content = re.sub(r'^```\n.*?\n```\n\n', '', content, flags=re.DOTALL)
     ```
   - **Extract frontmatter** (regex match on `---\n...\n---\n`):
     ```python
     fm_match = re.match(r'^---\s*\n(.*?)\n---\s*\n', content, re.DOTALL)
     fm_text  = fm_match.group(1)
     body     = content[fm_match.end():]
     ```
   - **Parse dates and sections** from frontmatter using regex:
     ```python
     all_dates  = re.findall(r'- (\d{4}-\d{2}-\d{2})', fm_text)
     left_secs  = re.findall(r'- (diary|almanac|cover|title-page|cash-account|bills-payable|address-list|blank)',
                             re.search(r'left:.*?(?=right:)', fm_text, re.DOTALL).group(0))
     right_secs = re.findall(r'- (diary|almanac|cover|title-page|cash-account|bills-payable|address-list|blank)',
                             re.search(r'right:.*', fm_text, re.DOTALL).group(0))
     ```
   - **Split body** into left and right page Markdown:
     ```python
     left_match  = re.search(r'^## Left Page\n(.*?)(?=^## Right Page|\Z)',
                             body, re.MULTILINE | re.DOTALL)
     right_match = re.search(r'^## Right Page\n(.*)', body, re.MULTILINE | re.DOTALL)
     left_md  = left_match.group(1).strip()  if left_match  else ''
     right_md = right_match.group(1).strip() if right_match else ''
     ```
   - Write `site/public/data/transcriptions/YYYY/STEM.json`
   - If either page contains `diary` section, write `pagefind-source/YYYY/STEM.html`

3. Build and write `manifest.json`, `year-index.json`, `date-index.json` to
   `site/public/data/`.

**CLI flags for partial builds (for local development):**
```bash
python build_site.py --year 1865
python build_site.py --years 1865 1866 1873
python build_site.py            # full corpus (default)
```

### Step 2: `npm run build` (Vite)

Vite bundles `site/js/main.js` and `site/css/main.css`, outputs hashed assets
to `dist/assets/`, copies `site/public/*` to `dist/`, and copies HTML files
to `dist/`.

**Key:** Because generated data is in `site/public/data/`, Vite copies it to
`dist/data/` automatically. No build order conflict.

### Step 3: Pagefind

```bash
npx pagefind --site pagefind-source --output-path dist/pagefind
```

`pagefind-source/` is generated by the Python build step and is never deployed.
Only `dist/pagefind/` goes to S3.

## 4. Pagefind Integration

### The SPA Problem

Pagefind indexes static HTML files. The viewer is a single-page app — Pagefind
cannot index dynamically rendered content. The solution: generate a parallel
set of thin HTML stubs for Pagefind to crawl, with search result URLs
configured via `data-pagefind-meta`.

### Pagefind Source Stubs (`pagefind-source/YYYY/STEM.html`)

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>January 1–2, 1865</title>
</head>
<body>
  <article data-pagefind-body data-pagefind-meta="url:/1865/0003">
    January, Sunday 1, 1865

    Painting on little oil sketch for Miss Sarah Marten until 2.30 o'clk.
    Then started from Studio and made 21 calls. Clear, cold, windy day.

    Monday 2.

    Henry M. Rutter died of Consumption — His death was very quiet...
  </article>
</body>
</html>
```

**Content rules:**
- Include **only diary entry text** (sections where type is `diary`)
- Strip all Markdown syntax: `#` headings → plain text, `**bold**` → plain,
  `[illegible]` markers preserved (useful search signal)
- Omit tables entirely (almanac/cash-account data adds noise)
- Omit non-diary pages (covers, blanks, title pages) — no stub generated
- **`data-pagefind-meta="url:/1865/0003"`** sets the result URL directly as a
  literal value, so search results link to the correct SPA route

**Estimated index size:** ~13 MB of plain text at full corpus → ~1.5–2 MB of
Pagefind index chunks, loaded on demand.

## 5. Frontend Architecture

### 5a. Startup Data Loading

Three small files fetched in parallel on page load:

```javascript
const [manifest, yearIndex, dateIndex] = await Promise.all([
  fetch('/data/manifest.json').then(r => r.json()),
  fetch('/data/year-index.json').then(r => r.json()),
  fetch('/data/date-index.json').then(r => r.json()),
])
```

Combined gzipped size: ~333 KB. Held in module-level variables for the
lifetime of the page. All navigation lookups after this are synchronous
in-memory operations.

The manifest is an object (`{ id: fileUri }`), so lookups by ID are O(1).
For sequential navigation (prev/next) and the scrubber, derive an ordered
array of IDs and a reverse lookup map at startup:

```javascript
const ids = Object.keys(manifest)  // V8 preserves insertion order for string keys
const idToIndex = new Map(ids.map((id, i) => [id, i]))  // O(1) reverse lookup
```

### 5b. Client-Side Routing

**URL pattern:** `/YYYY/NNNN`

```javascript
function urlToId(pathname) {
  const m = pathname.match(/^\/(\d{4})\/(\d{4})$/)
  if (m) {
    const id = `${m[1]}/${m[2]}`
    if (manifest[id]) return id  // valid ID
  }
  return null  // invalid URL
}

function navigate(id) {
  if (!id || !manifest[id]) {
    // Invalid ID — redirect to first image
    id = ids[0]
    history.replaceState({ id }, '', `/${id}`)
  } else {
    history.pushState({ id }, '', `/${id}`)
  }
  renderEntry(id)
}

window.addEventListener('popstate', e => navigate(e.state?.id ?? ids[0]))
```

**CloudFront SPA routing:** Custom error responses map HTTP 403 and 404 →
`index.html` with status 200. This allows direct access to `/1865/0001` etc.

### 5c. `viewer.js` — Rendering an Entry

The image and transcription load independently. The image loads immediately;
the transcription area shows a spinner until the fetch resolves.

**Race condition handling:** A navigation counter ensures that stale responses
from earlier navigations don't overwrite the current view.

```javascript
let currentNavId = 0  // incremented on each navigation

async function renderEntry(id) {
  const thisNavId = ++currentNavId
  const fileUri = manifest[id]
  const [year]  = id.split('/')
  const seq     = parseInt(id.split('/')[1])

  // 1. Update page title
  document.title = `${year} · ${seq} — The Diaries of James David Smillie`

  // 2. Image — loads immediately, no waiting on transcription
  img.src = `/images/${year}/${fileUri}.jpg`
  img.alt = `Diary photograph, ${year}, image ${seq}`

  // 3. Nav indicator
  const { count } = yearIndex[year]
  navLabel.textContent = `${year} · ${seq} of ${count}`

  // 4. Show spinner in transcription area while fetch is in-flight
  transcriptionArea.innerHTML = `
    <div class="tx-loading" aria-label="Loading transcription" aria-live="polite">
      <span class="spinner"></span>
    </div>`

  try {
    const tx = await fetch(`/data/transcriptions/${year}/${fileUri}.json`)
      .then(r => { if (!r.ok) throw r.status; return r.json() })

    // Guard: if user navigated away, discard this stale response
    if (thisNavId !== currentNavId) return

    const hasDiary = tx.sections.left.includes('diary') ||
                     tx.sections.right.includes('diary')

    // Update page title with date if available
    if (hasDiary && tx.dates.length) {
      document.title = `${formatDateRange(tx.dates)} — The Diaries of James David Smillie`
    }

    transcriptionArea.innerHTML = `
      <div class="date-label ${hasDiary ? '' : 'hidden'}">
        ${hasDiary ? formatDateRange(tx.dates) : ''}
      </div>
      <div class="transcription-columns">
        <div class="transcription-col">
          <div class="transcription-col-label">Left page</div>
          <div class="transcription-text">${marked.parse(tx.left || '')}</div>
        </div>
        <div class="transcription-col">
          <div class="transcription-col-label">Right page</div>
          <div class="transcription-text">${marked.parse(tx.right || '')}</div>
        </div>
      </div>`

  } catch (status) {
    if (thisNavId !== currentNavId) return  // stale
    const msg = status === 404
      ? 'Transcription not yet available.'
      : 'Could not load transcription.'
    transcriptionArea.innerHTML = `<p class="tx-placeholder">${msg}</p>`
  }

  // 5. Sync controls
  const index = idToIndex.get(id)
  scrubberInput.value = String(index)
  updateNavButtons(index)

  // 6. Prefetch adjacent images for snappier navigation
  prefetchAdjacentImages(index)
}

function prefetchAdjacentImages(index) {
  const prefetch = (i) => {
    if (i >= 0 && i < ids.length) {
      const id = ids[i]
      const [year] = id.split('/')
      const link = document.createElement('link')
      link.rel = 'prefetch'
      link.href = `/images/${year}/${manifest[id]}.jpg`
      document.head.appendChild(link)
    }
  }
  prefetch(index - 1)
  prefetch(index + 1)
}
```

The spinner is a pure-CSS animated ring — no image assets needed. The
transcription area has a `min-height` set so the page does not reflow
dramatically when the content loads.

### 5e. `nav.js`

```javascript
let currentIndex = 0

function navigateBy(delta) {
  const next = Math.max(0, Math.min(ids.length - 1, currentIndex + delta))
  if (next !== currentIndex) {
    currentIndex = next
    navigate(ids[next])
  }
}

prevBtn.addEventListener('click', () => navigateBy(-1))
nextBtn.addEventListener('click', () => navigateBy(+1))

function isInputFocused() {
  const tag = document.activeElement?.tagName
  return tag === 'INPUT' || tag === 'TEXTAREA'
}

document.addEventListener('keydown', e => {
  if (isInputFocused()) return
  if (e.key === 'ArrowLeft')  navigateBy(-1)
  if (e.key === 'ArrowRight') navigateBy(+1)
})

function updateNavButtons(index) {
  currentIndex = index
  prevBtn.disabled = index === 0
  nextBtn.disabled = index === ids.length - 1
}
```

### 5g. `datepicker.js`

```javascript
const sortedDates = Object.keys(dateIndex).sort()
datePicker.min = sortedDates[0]
datePicker.max = sortedDates.at(-1)

datePicker.addEventListener('change', e => {
  const date = e.target.value
  const id = dateIndex[date] ?? nearestDate(date)
  navigate(id)
})

function nearestDate(target) {
  // Binary search to find insertion point
  let lo = 0, hi = sortedDates.length - 1
  while (lo < hi) {
    const mid = (lo + hi) >> 1
    if (sortedDates[mid] < target) lo = mid + 1
    else hi = mid
  }
  const after  = sortedDates[lo]
  const before = sortedDates[lo - 1]
  if (!before) return dateIndex[after]
  if (!after)  return dateIndex[before]

  // Compare distances using Date timestamps (not string arithmetic)
  const targetTime = new Date(target).getTime()
  const beforeTime = new Date(before).getTime()
  const afterTime  = new Date(after).getTime()
  return (targetTime - beforeTime) <= (afterTime - targetTime)
    ? dateIndex[before]
    : dateIndex[after]
}
```

### 5h. `search.js`

Pagefind is loaded lazily when the user first interacts with the search input:

```javascript
let pagefind = null

async function getPagefind() {
  if (!pagefind) {
    pagefind = await import('/pagefind/pagefind.js')
    await pagefind.init()
  }
  return pagefind
}

searchInput.addEventListener('input', debounce(async e => {
  const query = e.target.value.trim()
  if (query.length < 2) { hideResults(); return }

  const pf = await getPagefind()
  const { results } = await pf.search(query)
  const data = await Promise.all(results.slice(0, 20).map(r => r.data()))
  renderResults(data)
}, 200))

function renderResults(data) {
  resultsPanel.innerHTML = data.map(r => `
    <a class="search-result" href="${r.url}">
      <span class="result-meta">${r.meta.title ?? ''}</span>
      <span class="result-excerpt">${r.excerpt}</span>
    </a>
  `).join('')
  resultsPanel.hidden = false

  resultsPanel.querySelectorAll('.search-result').forEach(a => {
    a.addEventListener('click', e => {
      e.preventDefault()
      const id = a.getAttribute('href').replace(/^\//, '')
      navigate(id)
      hideResults()
    })
  })
}
```

## 8. S3 + CloudFront Configuration

### S3 Bucket

Private bucket (no static website hosting). Access via CloudFront OAC only.

| Prefix | Cache-Control |
|---|---|
| `/*.html` | `no-cache` |
| `/assets/` | `public, max-age=31536000, immutable` |
| `/data/manifest.json`, `/data/year-index.json`, `/data/date-index.json` | `public, max-age=86400` |
| `/data/transcriptions/` | `public, max-age=2592000` |
| `/pagefind/` | `public, max-age=2592000` |
| `/images/` | `public, max-age=31536000, immutable` |

### CloudFront Distribution

- **Origin:** S3 via OAC
- **Default root object:** `index.html`
- **Custom error responses:**
  - 403 → `/index.html` (200)
  - 404 → `/index.html` (200)
- **Compression:** Gzip + Brotli enabled (manifest.json: 329 KB → ~131 KB)
- **SSL:** ACM certificate for `smilliediaries.org`
- **`www` redirect:** CloudFront function redirecting `www` → apex domain

**Note on `/about` routing:** The About page is served as `about.html` (flat
file, not `about/index.html`). Requests for `/about` will 403 → `index.html`,
so the SPA router must detect `/about` and redirect to `/about.html`. Or: add
a CloudFront Function on viewer-request to rewrite `/about` → `/about.html`.

## 12. Security Note

The `marked.parse()` function renders Markdown to HTML without sanitization.
Since transcription content comes from our own build pipeline (not user input),
this is acceptable. However, the Markdown files are generated by an LLM, so
there is a theoretical risk of HTML injection if the LLM were to output
malicious content. For defense in depth, consider configuring `marked` with
a custom renderer that escapes raw HTML, or use a sanitizer like DOMPurify on
the output. This is low priority given the controlled data source.
